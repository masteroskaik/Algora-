Règle ABSOLUE du professeur

⚠ À ne JAMAIS oublier sous peine de zéro ! L'affectation s'écrit obligatoirement := et jamais =. Cette règle est non-négociable dans tous les algorithmes du cours.
Pourquoi cette règle ?

En algorithmique formelle (pseudo-code du cours), le signe = est réservé aux comparaisons (ex : si a = b alors), tandis que := est l'opérateur d'affectation (on donne une valeur à une variable). Confondre les deux est une erreur conceptuelle grave.

✅ Correct — affectation 
age := 2026 - naissance;
somme := a + b;
i := 1;
❌ Interdit — zéro direct 
age = 2026 - naissance;   // FAUX
somme = a + b;            // FAUX
i = 1;                    // FAUX
Règle mnémotechnique : := se lit "reçoit" : « la variable reçoit la valeur ». Ex : « age reçoit 2026 moins naissance ».

Chapitre 1 – Rappel et base de la complexité algorithmique

1.1 Composants élémentaires d'un algorithme

Qu'est-ce qu'un algorithme ?

Un algorithme est une suite finie et ordonnée d'instructions permettant de résoudre un problème. Il se compose de primitives de base : affectation, lecture/écriture, conditions et boucles. Chaque instruction est considérée comme une opération élémentaire qui prend un temps constant.

InstructionSyntaxe exacteExemple

Boucle tantquetantque cond faire … finfaire;Condition testée avant chaque itération
Boucle pour (croissant)pour i := 1 haut n faire … finfaire;i = 1, 2, …, n
Boucle pour (décroissant)pour i := n bas 1 faire … finfaire;i = n, …, 2, 1
Boucle répéterrépéter … jusqu'à condition;Condition testée après → exécutée au moins 1 fois
Différence entre les 3 types de boucles
  • tantque : on teste la condition avant d'entrer. Si la condition est fausse dès le départ, le corps n'est jamais exécuté.
  • pour : variante simplifiée de tantque avec un compteur. haut = on incrémente, bas = on décrémente.
  • répéter…jusqu'à : on teste la condition après. Le corps est exécuté au moins une fois, même si la condition est d'emblée vraie.

// Structure conditionnelle complète (avec sinon)
si condition alors

1.2 Fonctions et Procédures – Syntaxe complète

Fonction vs Procédure — quelle différence ?
  • Une fonction retourne une valeur (type non vide). Elle s'encadre avec debfonc / finfonc et utilise retourner.
  • Une procédure ne retourne rien (type vide). Elle s'encadre avec debproc / finproc.

Types de paramètres : d = donnée (entrée) r = résultat (sortie) dr = donnée-résultat (entrée+sortie)

Explication des modes de passage de paramètres
  • d (donnée) : le paramètre est fourni à la fonction, elle peut le lire mais pas le modifier à l'extérieur. Équivaut à un passage par valeur.
  • r (résultat) : la fonction doit écrire une valeur dans ce paramètre. Il est vide à l'entrée, rempli à la sortie.
  • dr (donnée-résultat) : la fonction lit ET modifie le paramètre. Équivaut à un passage par référence.
// Exemple de FONCTION (retourne un entier)
entier calculAge(d entier an, d entier naiss)
debfonc
  entier age;
  age := an - naiss;
  retourner age;
finfonc;

// Appel de la fonction
écrire("Vous avez " & calculAge(annee, naissance) & " ans");

// Exemple de PROCÉDURE (ne retourne rien, modifie les paramètres)
vide permut(dr entier a, dr entier b)
debproc
  entier tmp;
  tmp := a;  // sauvegarde a
  a := b;    // a reçoit la valeur de b
  b := tmp;  // b reçoit l'ancienne valeur de a
finproc;
// a et b sont lus ET modifiés → paramètres dr

1.3 Notion de complexité

Qu'est-ce que la complexité ?

La complexité d'un algorithme mesure ses ressources consommées en fonction de la taille $n$ des données en entrée. On distingue deux types :

  • Complexité spatiale : l'espace mémoire utilisé (combien de variables, tableaux…)
  • Complexité temporelle : le nombre d'opérations élémentaires exécutées. Chaque opération de base (affectation, addition, comparaison) compte pour 1 unité de temps.

On s'intéresse surtout à la complexité dans le pire cas (notation $O$, lire « grand O »), car elle garantit une borne maximale sur le temps d'exécution.

Exemple de calcul pas à pas : a := b + 3;
→ 1 addition (b + 3) + 1 affectation (a :=) = 2 unités de temps
écrire("Résultat = " & a); → 1 unité de temps

Complexité d'une boucle

Comment compter les itérations d'une boucle ?

Une boucle pour i := 1 haut n faire exécute son corps exactement $n$ fois. Si le corps contient $k$ opérations, la boucle coûte $n \times k$ opérations. Deux boucles imbriquées de taille $n$ coûtent $n \times n = n^2$ opérations.

Complexité d'un conditionnel

TypeCas favorableCas défavorableExplication
si cond alors I finsi$T = T_{cond}$$T = T_{cond} + T_I$Favorable = condition fausse, on ne fait que tester
si cond alors I1 sinon I2 finsi$T = T_{cond} + \min(T_{I1}, T_{I2})$$T = T_{cond} + \max(T_{I1}, T_{I2})$On exécute la plus courte ou la plus longue branche
Exemple concret : 
si i=0 alors                // 1 opération (comparaison)
    debut := nouveau(joueur); // 1 opération (affectation)
    p := debut;              // 1 opération (affectation)
finsi;
// Cas défavorable : T = 1 (condition) + 2 (instructions) = 3 = O(1)

1.4 Ordres de grandeur – À mémoriser absolument

La notation O( ) — ce qu'elle signifie vraiment

$O(f(n))$ signifie que le temps d'exécution ne dépasse pas $c \times f(n)$ pour un certain constante $c$ et pour $n$ suffisamment grand. On garde uniquement le terme dominant et on ignore les constantes. Exemples : $T = 5n + 3 = O(n)$ ; $T = 2n^2 + 100n = O(n^2)$.

NotationNomIntuitionExemple typique
$O(1)$ConstanteToujours la même vitesse, quelle que soit la tailleAccès direct à t[i]
$O(\log n)$LogarithmiqueOn divise l'espace par 2 à chaque étapeRecherche dichotomique
$O(n)$Linéaire1 boucle simple sur n élémentsParcours tableau, liste
$O(n \log n)$Quasi-linéaireDiviser le problème puis recombinerQuicksort, Tri fusion
$O(n^2)$Quadratique2 boucles imbriquées de taille nTri insertion (pire cas)
$O(n^3)$Cubique3 boucles imbriquées de taille nMultiplication de matrices naïve
$O(2^n)$ExponentielleTester tous les sous-ensemblesFibonacci récursif naïf
$O(n!)$FactorielleTester toutes les permutationsVoyageur de commerce (brute-force)
Ordre croissant — du plus rapide au plus lent : $$O(1) < O(\log n) < O(n) < O(n\log n) < O(n^2)$$ $$ O(n^2) < O(n^3) < O(2^n) < O(n!)$$
Comprendre l'impact concret des complexités

Pour $n = 1\,000$ éléments, voici le nombre d'opérations approximatif :

  • $O(1)$ → 1 opération
  • $O(\log n)$ → ≈ 10 opérations
  • $O(n)$ → 1 000 opérations
  • $O(n^2)$ → 1 000 000 opérations
  • $O(2^n)$ → plus d'opérations qu'il y a d'atomes dans l'univers !

C'est pourquoi un algorithme $O(n^2)$ devient inutilisable pour de grandes données, tandis qu'un $O(n \log n)$ reste efficace.

Chapitre 2 – Tableaux et Listes chaînées

2.1 Tableaux – Structure et parcours

Qu'est-ce qu'un tableau ?

Un tableau est une collection d'éléments du même type, stockés de manière contigüe en mémoire, accessibles via un indice. L'accès à l'élément t[i] est toujours $O(1)$ car l'adresse est calculée directement.

Le parcours complet d'un tableau de $n$ éléments est toujours $O(n)$ : on visite chaque élément exactement une fois.

// Parcours itératif avec tantque — T(n) = 1 + 3n = O(n)
vide parcours(d entier t[], d entier n)
debfonc
  entier i;
  i := 1;                    // 1 opération (init)
  tantque i≤n faire          // n fois : 1 comparaison
    traiter(t[i]);           // 1 opération
    i := i + 1;             // 2 opérations (addition + affectation)
  finfaire;
finfonc;
// Total : 1 + n×(1+1+2) = 1 + 4n → O(n)

// Forme abrégée avec pour — équivalent O(n)
pour i := 1 haut n faire traiter(t[i]); finfaire;

2.2 Recherches dans un tableau

Recherche séquentielle

Pour un tableau non trié. On parcourt un à un jusqu'à trouver l'élément ou atteindre la fin.
Complexité : $O(n)$
Pire cas : élément absent ou en dernière position.

Recherche dichotomique

Pour un tableau obligatoirement trié. On coupe le tableau en deux à chaque étape.
Complexité : $O(\log_2 n)$
Très efficace pour les grands tableaux.

Comment fonctionne la dichotomie ?

On maintient deux bornes inf et sup. À chaque étape :

  • On calcule le milieu : m := (inf + sup) div 2
  • Si t[m] = elem → trouvé !
  • Si t[m] < elem → l'élément est dans la moitié droite → inf := m + 1
  • Sinon → l'élément est dans la moitié gauche → sup := m - 1

Après $q$ étapes, on a $n / 2^q = 1$, donc $q = \log_2(n)$. D'où la complexité $O(\log_2 n)$.

booléen dich(d entier t[], d entier n, d entier elem)
debfonc
  entier inf, sup, m; booléen trouve;
  trouve := faux;
  si (elem≥t[1]) et (elem≤t[n]) alors
    inf := 1; sup := n;
    tantque (inf≤sup) et (non trouve) faire
      m := (inf+sup) div 2;
      si t[m]=elem alors trouve := vrai;
      sinon si t[m]<elem alors inf := m+1;
               sinon sup := m-1;
      finsi; finsi;
    finfaire;
  finsi;
  retourner trouve;
finfonc;

Preuve de la complexité :

$$\frac{n}{2^q} = 1 \implies 2^q = n \implies q = \log_2(n)$$ $$\implies T(n) = O(\log_2 n)$$

2.3 Algorithmes de tri – Comparaison

Pourquoi trier ?

Trier un tableau permet de retrouver des éléments plus rapidement (recherche dichotomique en $O(\log n)$ au lieu de $O(n)$). Le coût du tri est une investissement si on effectue de nombreuses recherches ensuite.

TriPrincipeComplexitéRemarque
InsertionInsérer chaque élément à sa bonne place dans la partie déjà triée$O(n^2)$ pire cas
$O(n)$ meilleur cas		Efficace pour petits tableaux ou tableaux quasi-triés
QuicksortDiviser pour régner : choisir un pivot, partitionner, récursion$O(n \log n)$ moyen
$O(n^2)$ pire		Pivot = t[inf] par défaut dans ce cours
Tri FusionDiviser en 2, trier chaque moitié récursivement, puis fusionner$O(n \log n)$ toujoursStable, prévisible, mais utilise de la mémoire supplémentaire
Comprendre le Quicksort — Diviser pour régner

L'idée centrale : choisir un pivot (valeur de référence), puis réorganiser le tableau en deux parties :

  • Tous les éléments ≤ pivot à gauche
  • Tous les éléments > pivot à droite

Après cette opération (appelée segmentation), le pivot est à sa position définitive. On applique ensuite récursivement le même principe aux deux sous-tableaux.

vide trisegmentation(dr entier t[], d entier inf, d entier sup)
debfonc
  entier place;
  si inf < sup alors
    segmentation(t, inf, sup, place); // pivot mis à "place"
    trisegmentation(t, inf, place-1); // tri partie gauche
    trisegmentation(t, place+1, sup); // tri partie droite
  finsi;
finfonc;
// Appel initial : trisegmentation(t, 1, n)

2.4 Listes linéaires chaînées

Tableaux vs Listes chaînées — quand utiliser quoi ?
  • Tableau : accès direct en $O(1)$, mais taille fixe et insertions/suppressions coûteuses si on déplace des éléments.
  • Liste chaînée : taille dynamique, insertion en tête en $O(1)$, mais accès séquentiel uniquement ($O(n)$ pour atteindre l'élément $k$).

Chaque cellule contient : info (valeur) + suivant (pointeur vers la cellule suivante). La dernière cellule pointe vers nil.

Visualiser une liste chaînée

Imaginez des wagons de train : chaque wagon (cellule) contient une marchandise (info) et est attaché au wagon suivant (suivant). Le dernier wagon n'est attaché à rien (nil). Pour aller au wagon 50, il faut partir du wagon 1 et avancer un par un → $O(n)$.

// Création d'une cellule et insertion en TÊTE — O(1)
pointeur liste, p;
nouveau(p);              // alloue une nouvelle cellule en mémoire
p->info := valeur;     // on met la valeur dans la nouvelle cellule
p->suivant := liste;  // la nouvelle cellule pointe vers l'ancienne tête
liste := p;           // la nouvelle cellule devient la tête
OpérationComplexitéExplication
Parcours complet$O(n)$On doit visiter chaque cellule une à une
Insertion en tête$O(1)$On ne modifie que le premier pointeur, pas besoin de parcourir
Insertion en fin$O(n)$Il faut d'abord parcourir toute la liste pour atteindre la dernière cellule
Recherche d'une valeur$O(n)$Pas d'accès direct : parcours séquentiel obligatoire

Chapitre 3 – Piles et Files d'attente

3.1 Pile (Stack) – LIFO

LIFO = Last In First Out — « Dernier entré, premier sorti »
Analogie : une pile d'assiettes. On pose toujours la nouvelle assiette sur le dessus, et on prend toujours l'assiette du dessus. Impossible d'accéder directement aux assiettes du bas.
Pourquoi utiliser une pile ?

Les piles sont fondamentales en informatique : elles gèrent les appels de fonctions récursives (la pile d'exécution du processeur), permettent d'implémenter le bouton Annuler (Ctrl+Z), et servent à évaluer des expressions mathématiques (forme postfixée).

PrimitiveRôleComplexité
empiler(val)Ajoute val au sommet de la pile$O(1)$
dépiler(val)Retire l'élément du sommet dans val$O(1)$
sommetpile()Consulte le sommet sans le retirer$O(1)$
pilevide()Retourne vrai si la pile est vide$O(1)$
pilepleine()Retourne vrai si la pile est pleine (représentation contiguë uniquement)$O(1)$
initpilevide()Initialise une pile vide$O(1)$
Représentation contiguë (tableau)

Un tableau + un entier sommet qui indique l'indice du dernier élément inséré. Simple à coder, mais la taille est fixe à la déclaration.

Représentation chaînée (liste)

Une liste chaînée dont la tête est le sommet. Taille dynamique (pas de limite). La pile est vide quand pile = nil.

3.2 Traitement des expressions arithmétiques

Pourquoi différentes notations ?

La notation habituelle (infixée, ex: A + B) nécessite des parenthèses et des règles de priorité. Les notations préfixée et postfixée n'ont besoin d'aucune parenthèse : l'ordre de lecture suffit à lever toute ambiguïté. C'est pourquoi les compilateurs et calculatrices utilisent souvent la forme postfixée en interne.

FormeSigleExemple pour ((A-B)+C)Ordre des opérandes
Complètement parenthéséeexcp((A-B)+C)Notation sans ambiguïté
Préfixéeexpref+-ABCOpérateur AVANT ses opérandes
PostfixéeexpostAB-C+Opérateur APRÈS ses opérandes
Infixée—A-B+CNotation habituelle (entre opérandes)
Mnémotechnique :
Préfixée = opérateur avant (précède les opérandes)
Postfixée = opérateur après (suit les opérandes)
Infixée = opérateur entre les opérandes — notre écriture naturelle

Évaluation d'une expression postfixée avec une pile

Algorithme d'évaluation

On lit l'expression de gauche à droite, caractère par caractère :

  • Si c'est une variable/nombre → on l'empile
  • Si c'est un opérateur binaire (ex: +, -, *, /) → on dépile les 2 derniers opérandes, on effectue le calcul, on empile le résultat
  • Si c'est le symbole de fin (#) → le résultat final est au sommet de la pile
Exemple complet : Évaluer AB*CD+/ avec A=20, B=4, C=9, D=7
A→empile 20 · B→empile 4 · *→dépile 4 et 20, calcule 20×4=80, empile 80
C→empile 9 · D→empile 7 · +→dépile 7 et 9, calcule 9+7=16, empile 16
/→dépile 16 et 80, calcule 80÷16=5, empile 5 · #→résultat = 5

3.3 File d'attente (Queue) – FIFO

FIFO = First In First Out — « Premier entré, premier sorti »
Analogie : une file au guichet ou une file de supermarché. Le premier arrivé est le premier servi. On entre par l'arrière (queue) et on sort par l'avant (tête).
Applications des files d'attente
  • File d'impression : les documents sont imprimés dans l'ordre d'envoi
  • BFS (Parcours en largeur) : algorithme de graphes qui explore les voisins niveau par niveau
  • Gestion de processus : le système d'exploitation attribue le processeur aux processus dans l'ordre
  • Buffers réseau : les paquets de données sont traités dans l'ordre d'arrivée
  • Insertion (enfiler) : toujours en fin de file (côté queue)
  • Suppression (défiler) : toujours en tête de file
ReprésentationAvantageInconvénient
Simple (tableau)Facile à coderGaspillage mémoire : la tête avance mais l'espace libéré n'est pas réutilisé
Avec décalageÉconomise la mémoireDécalages coûteux $O(n)$ à chaque suppression
CirculaireOptimal : $O(1)$ pour toutes les opérationsPlus complexe à implémenter
ChaînéeTaille dynamique, $O(1)$ pour enfiler et défilerSurcoût mémoire pour les pointeurs

Réponse attendue

$T(n) = O(\log_2 n)$

Justification : à chaque itération, on divise la taille du sous-tableau par 2. Après $q$ itérations :

5
Calculer la complexité de : pour i:=1 haut n faire pour j:=1 haut n faire a:=a+1; finfaire; finfaire;
Voir
Réponse attendue

La boucle externe s'exécute $n$ fois. Pour chaque itération externe, la boucle interne s'exécute $n$ fois. L'instruction a := a+1 (2 opérations) s'exécute $n \times n = n^2$ fois.

$$T(n) = n \times n \times 2 = 2n^2 = O(n^2)$$

Règle générale : $k$ boucles imbriquées de taille $n$ donnent $O(n^k)$.

7
Expliquer le principe du Quicksort. Quelle est sa complexité dans le cas moyen ?
Voir
Réponse attendue

Principe — Diviser pour régner :

  1. Choisir un pivot (ex : t[inf])
  2. Partitionner : éléments ≤ pivot à gauche, éléments > pivot à droite
  3. Le pivot est à sa position définitive
  4. Récursion sur les deux sous-tableaux

Complexités :

  • Cas moyen : $O(n \log n)$ — le pivot coupe approximativement en deux parties égales
  • Pire cas : $O(n^2)$ — tableau déjà trié avec pivot = premier élément (pas de partitionnement)

Avec sinon, cas défav. = $T_{cond}+\max(T_{I1},T_{I2})$
